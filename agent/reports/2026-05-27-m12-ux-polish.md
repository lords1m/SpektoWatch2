# M12 UX polish — 2026-05-27

Two ad-hoc UX fixes landed on `redesign/liquid-glass` after a real-device
session surfaced regressions in the redesign. Both fall under M12's
"Stability & Polish" remit (task-8, already completed) — recorded here
rather than reopening task-8.

## 1. SingleValueWidget vertical height

### Observed

On device, single-value widgets at 1×1 were ~200pt tall with a 36pt
number floating in the middle and large dead space below. Felt
disproportionate next to the column width of ~122pt (rough 1.6:1
portrait ratio for a single digit).

### Root cause

`WidgetSize.height` returned a fixed 200pt baseline per row regardless
of widget type. The baseline is correct for chart-style widgets
(spectrogram, waterfall, frequency display) but wasteful for compact
value readouts.

### Fix

Per-type row baseline introduced via a new static helper +
computed property on `WidgetConfiguration`:

- `WidgetConfiguration.baseRowHeight(for type:)` — returns **110pt**
  for `.singleValue`, `.levelMeter`, `.phaseMeter`; **200pt** for
  everything else.
- `WidgetConfiguration.frameHeight` — computes total frame height
  from the type's base × rows + 12pt inter-row spacing.

`WidgetCardView` now calls `widget.frameHeight` instead of
`widget.size.height` at both call sites (`kernelHeight` floor + outer
`.frame(height:)`). The existing `WidgetSize.height` accessor is
preserved for any external callers but is no longer authoritative.

### Effect

- 1×1 single-value cells drop from 200pt → 110pt (-45%).
- 2×2 single-value cells drop from 412pt → 232pt (-44%).
- Chart-style widgets unchanged.
- Header strip + chrome overhead unchanged; the kernel still respects
  the 60pt floor.

### Open

110pt is a first guess. If on-device feedback says still too tall or
now too cramped, the value is one line to tune.

## 2. Preset-rail highlight didn't follow TabView swipes

### Observed

Swiping between layouts via `TabView` did not update the highlighted
chip in `PresetRailView`. The rail kept the previously selected preset
highlighted even though the dashboard had moved to a different
preset's layout.

### Root cause

Two independent state sources:

- `dashboardManager.activeLayoutIndex` (`@Published`) — drives the
  `TabView` selection and the layout-name display.
- `@AppStorage("dashboard.activePreset")` (in `ModularDashboardView`)
  — drives the chip highlight in `PresetRailView`.

Tapping a chip wrote to both (`onSelect` set `activeID` AND called
`applyPreset`, which advanced `activeLayoutIndex`). Swiping the
TabView only changed the first; the AppStorage value was never
touched, so the highlight stayed stale.

### Fix

`ModularDashboardView` gained a small helper
`syncActivePresetFromLayout()` that reads
`dashboardManager.currentLayoutName`, strips the `"Preset: "` prefix
(the format `DashboardManager.applyPreset(id:)` writes), and writes
the resulting ID into `activePresetID` if it differs.

The helper is invoked via:
- `.onChange(of: dashboardManager.activeLayoutIndex)` — every swipe
  / programmatic change.
- `.onAppear` — initial sync after cold launch.

User-created layouts not following the `"Preset: <id>"` convention
are skipped (the rail stays on its previous value, which is the
existing behavior for non-preset layouts).

### Effect

Swiping between two preset-derived layouts now animates the chip
highlight to follow.
